## What Is BESCOM Pulse?

BESCOM Pulse is a prototype AI copilot for grid operators at Bengaluru's electricity distribution company. As EV adoption accelerates, concentrated demand on the distribution network creates grid stress events — BESCOM Pulse helps operators manage this in real time.

**The system does three things:**

- **Monitors** 15 zones across Bengaluru, tracking EV charging load against feeder capacity
- **Optimises** demand using a 4-tier hierarchical algorithm (SoC cap → time-shift → spatial redirect → infra trigger)
- **Plans** by identifying optimal coordinates for new charging hubs before the grid breaks

**Privacy by design:** The MCP Server acts as a secure middleware — the AI only ever sees masked, aggregated data. No vehicle IDs, user details, or raw consumption data is exposed.

## Quick Start

### Prerequisites

- Python 3.9 or higher
- pip
- A terminal / Jupyter environment

### Step 1 — Clone / Download

Place all three files in the same directory:

```
bescom_pulse_masked_dataset.csv
bescom_pulse_mcp_server.ipynb
README.md
```

### Step 2 — Install Dependencies

```bash
pip install pandas numpy streamlit plotly anthropic python-dotenv scikit-learn
```

Or run **Cell 1** of the notebook, which installs everything automatically.

### Step 3 — Run the Notebook

Open `bescom_pulse_mcp_server.ipynb` in JupyterLab, Jupyter Notebook, or VS Code and run cells **top to bottom**:

```bash
jupyter notebook bescom_pulse_mcp_server.ipynb
# or
jupyter lab
```

Run cells in order: **Cell 1 → Cell 2 → ... → Cell 7**. Cell 7 writes `bescom_pulse_app.py` to your working directory.

### Step 4 — Launch the Streamlit UI

```bash
streamlit run bescom_pulse_app.py
```

Open your browser at **http://localhost:8501**

### Step 5 — (Optional) Enable Live AI Chat

```bash
export ANTHROPIC_API_KEY=your_api_key_here
streamlit run bescom_pulse_app.py
```

Without the API key, the chat interface uses a fast rule-based fallback that still answers zone queries accurately from the dataset.

| `anomaly_flag` | int | `1` = anomalous consumption pattern detected |

### Status Thresholds

```
load_pct < 70%   →  GREEN   (Tier 1 only — SoC cap recommended)
load_pct < 85%   →  YELLOW  (Tier 2 — time-shift incentive triggered)
load_pct < 95%   →  ORANGE  (Tier 3 — spatial redirect triggered)
load_pct ≥ 95%   →  RED     (Tier 4 — infrastructure expansion flagged)
```

### The 4-Tier Optimization Engine

The `BESCOMOptimizationEngine` class in **Cell 4** runs a cascade — each tier is evaluated in order and the highest triggered tier determines the operator action.

```
Charging Request / Grid Stress Event
            │
            ▼
┌─────────────────────────────┐
│  TIER 1: SoC Cap (60%)      │  ← Always active
│  Reduce charging window     │
│  Extend battery life        │
└──────────────┬──────────────┘
               │ load > 70%?
               ▼
┌─────────────────────────────┐
│  TIER 2: Time-Shift Pricing │
│  Nudge user to off-peak     │
│  Show ₹ savings             │
└──────────────┬──────────────┘
               │ load > 85%?
               ▼
┌─────────────────────────────┐
│  TIER 3: Spatial Redirect   │
│  Route to nearest Green Zone│
│  Haversine distance calc    │
└──────────────┬──────────────┘
               │ load > 95%?
               ▼
┌─────────────────────────────┐
│  TIER 4: Infra Trigger      │
│  Centre-of-mass calculation │
│  Recommend nodal point (x,y)│
│  Estimate CAPEX             │
└─────────────────────────────┘
```

**Key methods on `BESCOMOptimizationEngine`:**

```python
engine = BESCOMOptimizationEngine(df)

# Run the full cascade for a zone
result = engine.run_decision("Z001")

# Or call individual tiers
engine.tier1_soc_recommendation("Z001")   # → SoC cap advice
engine.tier2_time_shift_incentive("Z001") # → pricing comparison
engine.tier3_spatial_redirect("Z001")     # → nearest green zone
engine.tier4_infra_trigger("Z001")        # → nodal coordinates + CAPEX
```

## MCP Server Architecture

The `MCPToolHandler` in **Cell 5** is the secure middleware between the AI and the data. It exposes 7 tools:

| Tool Name | Description |
|-----------|-------------|
| `get_zone_status` | Current load, status, and tier for one zone |
| `get_all_zone_summary` | Aggregated summary of all 15 zones |
| `run_optimization` | Execute the full 4-tier cascade for a zone |
| `get_peak_analysis` | Peak vs off-peak demand comparison |
| `get_infra_recommendations` | All Tier 4 zones with nodal coordinates |
| `get_anomaly_report` | Zones with flagged anomalous patterns |
| `get_hourly_demand_trend` | Average load by hour of day |

### How the MCP Layer Enforces Privacy

```
Operator Question
      │
      ▼
  Claude (LLM)
      │  calls tool with parameters
      ▼
  MCPToolHandler.execute(tool_name, tool_input)
      │  queries only aggregated / masked fields
      │  never touches: masked_vehicle_id, raw meter reads
      ▼
  JSON result (anonymised)
      │
      ▼
  Claude composes answer
      │
      ▼
  Operator sees response
```

The raw `masked_vehicle_id` field exists only in the dataset for auditability — the MCP tools never include it in any response passed to the LLM.

### Calling a Tool Directly (Testing)

```python
handler = MCPToolHandler(df, engine)

# Get zone status
print(handler.execute("get_zone_status", {"zone_id": "Z001"}))

# Run optimization
print(handler.execute("run_optimization", {"zone_id": "Z003"}))

# Infrastructure recommendations
print(handler.execute("get_infra_recommendations", {}))
```

## Streamlit UI — Page Guide

After running **Cell 7** and launching Streamlit, you'll see 6 pages in the sidebar:

### 📊 Overview
The home dashboard. Shows KPI cards (total sessions, critical zones, infra triggers, anomalies), a colour-coded zone status table, a status distribution donut chart, and tier activation bar chart.

### 🗺️ Zone Heatmap
An interactive Mapbox scatter map of all 15 zones — colour encodes load %, bubble size encodes session volume. A second map shows proposed charging hub locations (star markers) computed from the centre-of-mass of Tier 4 demand events.

### ⚙️ Optimization Engine
Select any zone and see a live gauge showing current load %, followed by the 4 tier cards — each expands to show the full recommendation, pricing comparison (Tier 2), redirect target (Tier 3), or nodal coordinates with CAPEX estimate (Tier 4). The **Approve / Edit / Override** buttons at the bottom simulate the human-in-the-loop decision flow.

### 💬 MCP Chat Interface
A conversational interface powered by the MCP tools. Type any question about the grid — the backend either calls the Anthropic API (with tool use) or falls back to rule-based responses. Quick-query chips let you one-click common operator questions. Conversation history is maintained per session.

**Example queries that work:**
- *"Why is HSR Layout showing a red alert?"*
- *"Which zones need infrastructure expansion?"*
- *"Show me all zone status"*
- *"Explain the 60% SoC cap strategy"*
- *"Which zones have anomalies?"*

### 🏗️ Infrastructure Planning
The Tier 4 view. Shows all overloaded zones in a table with overload event counts, average load, proposed hub coordinates, estimated CAPEX, and priority rating. A horizontal bar chart ranks zones by CAPEX. The nodal map shows existing zones (coloured by status) alongside proposed hub locations (purple stars).

### 📈 Demand Analytics
Three sub-tabs: **Hourly Demand** shows load % by hour with tier threshold lines annotated; **Charger Mix** shows the distribution of charger types and their average EV load contribution; **SoC & Pricing** shows session start SoC histograms and a peak vs off-peak tariff comparison.

## AI Chat Integration

The chat layer in **Cell 6** uses the Anthropic API with native tool use. Here is the full flow:

```python
# 1. Tools are registered with the Anthropic client
anthropic_tools = [{ "name": "get_zone_status", ... }]

# 2. User message + conversation history sent to Claude
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    tools=anthropic_tools,
    messages=messages,
)

# 3. If Claude calls a tool, MCPToolHandler executes it
if response.stop_reason == "tool_use":
    tool_output = handler.execute(block.name, block.input)
    # Result appended and sent back to Claude

# 4. Claude composes the final answer
# 5. Conversation history maintained for multi-turn context
```

The system prompt instructs Claude to always cite zone name, load %, and tier; use ₹ for currency; frame the 60% SoC cap as a battery health benefit; and be concise because operators need to act quickly.

### Fallback Mode (No API Key)

Without `ANTHROPIC_API_KEY`, the chat uses the `fallback_response()` function which pattern-matches the query and calls the same MCP tool handler directly. Zone name lookups, infra queries, anomaly reports, and SoC explanations all work without any API call.

## Troubleshooting

**`ModuleNotFoundError: No module named 'streamlit'`**
Run `pip install streamlit plotly anthropic pandas numpy` and try again.

**`FileNotFoundError: bescom_pulse_masked_dataset.csv`**
Make sure the CSV is in the **same directory** as `bescom_pulse_app.py`. Both files must be co-located.

**`bescom_pulse_app.py not found`**
You need to run **Cell 7** of the notebook first. It writes this file to your working directory.

**Map shows blank / no tiles**
The zone heatmap uses Mapbox via Plotly's `carto-darkmatter` style, which does not require a token. If it still shows blank, check your internet connection — the map tiles are loaded from a CDN.

**Chat returns "I'm not sure"**
Without an `ANTHROPIC_API_KEY`, the fallback responds to zone names and keywords. Try mentioning a zone by name (e.g. "Whitefield") or a keyword like "infra", "red", "anomaly", or "soc".

**Port 8501 already in use**
```bash
streamlit run bescom_pulse_app.py --server.port 8502
```
